# 提交材料清单

## 1. 需要提交的三部分

### A. 分享链接

提交内容建议：

- 正式线上地址
- 保证可长期访问
- 最好绑定 GitHub 自动部署

不建议提交：

- `localhost`
- 临时隧道链接
- 可能过期的预览链接

推荐平台：

- 静态或前端项目：Vercel / Netlify
- 含后端接口项目：Render

### B. 代码

提交内容建议：

- GitHub 仓库地址
- `README.md`
- 必要运行说明
- 关键目录清晰

最低要求：

1. 仓库能看出项目结构
2. 有本地启动说明
3. 有功能简介
4. 有部署说明或部署计划

### C. 截图

建议放在：

- `docs/screenshots/`

建议最少准备以下截图：

1. 首页 / 模块入口
2. 首次测评页面
3. 职业匹配报告页面
4. 主副职业方向选择页面
5. 个人主页中的职业方向卡片

如果有增强功能，再补：

1. 趋势图页面
2. 提醒弹窗
3. 月报页面

## 2. 命名建议

截图文件建议统一命名：

- `01-home.png`
- `02-assessment.png`
- `03-report.png`
- `04-career-selection.png`
- `05-dashboard-card.png`

## 3. GitHub 同步建议

推荐流程：

1. 本地开发
2. 本地运行验证
3. 提交到 Git
4. 推送到 GitHub
5. 部署平台自动更新
6. 补充截图

## 4. 永久链接建议

如果你后续会一直更新，这里最关键的是：

- GitHub 用来托管代码
- 部署平台用来提供稳定访问地址

最稳的方式是：

1. 创建 GitHub 仓库
2. 把项目部署到 Vercel / Netlify / Render
3. 让部署平台绑定 GitHub 仓库主分支
4. 后续每次 push 到主分支时自动重新部署

这样你的分享链接就不会因为本机关机或网络变化而失效。

## 5. 推荐截图生成方式

如果项目是 Web 应用，建议后续使用自动化方式截图：

- Playwright 自动打开页面并截图
- 这样每次改版都能稳定生成同一套提交图

## 6. 提交前最终检查

提交前至少确认：

1. 分享链接可打开
2. GitHub 仓库可访问
3. 关键流程可跑通
4. 截图和当前版本一致
5. README 与实际运行方式一致
