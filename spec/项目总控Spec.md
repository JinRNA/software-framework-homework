# 项目总控 Spec

## 1. 项目定位

本项目是课程大作业，目标不是一次性做完全部 PRD，而是做出一个：

- 能本地运行
- 能线上访问
- 能持续迭代
- 能稳定提交

的人才素描模块。

## 2. 交付目标

最终至少形成三类独立提交材料：

1. 分享链接
2. 代码仓库
3. 系统截图

建议对应如下：

- 分享链接：正式部署地址，例如 Vercel / Netlify / Render 的生产域名
- 代码仓库：GitHub 仓库
- 截图：关键页面 PNG/JPG，统一放在 `docs/screenshots/`

## 3. 建议技术路线

如果你现在还没定技术栈，优先选择“开发快、部署稳、便于演示”的方案。

### 方案 A：前端优先，最快出结果

- 前端：React + Vite
- 数据：本地 mock 数据 / JSON
- 匹配算法：前端本地加权计算
- 部署：Vercel 或 Netlify

适合场景：

- 课程更看重产品展示、页面完整度、交互流程
- 当前时间有限，先做可演示版本

### 方案 B：前后端完整，更像真实系统

- 前端：React + Vite
- 后端：ASP.NET Core Web API
- 数据库：SQLite / SQL Server LocalDB
- 部署：前端 Vercel，后端 Render；或整体 Render

适合场景：

- 课程强调“软件开发框架”的完整工程实现
- 你希望把接口、数据模型、计算逻辑做得更扎实

## 4. MVP 范围

第一版不要试图覆盖整个 PRD，优先做最能演示价值的闭环。

### 必做功能

1. 用户填写首次测评问卷
2. 系统计算多个职业方向匹配度
3. 展示职业匹配报告
4. 用户选择 1 个主方向、最多 2 个副方向
5. 首页展示当前职业方向和匹配度

### 第二阶段功能

1. 匹配度趋势图
2. 手动触发“重新计算”
3. 行为变化提醒
4. 月报/年度报告页面

### 第三阶段功能

1. 职业标签库管理
2. 市场数据接入
3. 后台定时更新
4. 更复杂的推荐算法

## 5. 页面清单

建议最少做这 5 个页面/视图：

1. 欢迎页 / 模块首页
2. 首次测评页
3. 职业匹配报告页
4. 职业方向管理页
5. 个人主页中的“职业方向”卡片页

如果时间允许，再补：

1. 匹配度趋势详情页
2. 月报页
3. 管理端职业标签页

## 6. 最小数据模型

至少需要这些实体：

- 用户测评 `user_assessment`
- 职业匹配 `career_match`
- 用户职业选择 `user_career_choice`
- 职业标签库 `career_tags`

MVP 阶段允许：

- 用 JSON 文件或内存数据替代数据库
- 用固定职业样本替代完整职业库
- 用手工配置的市场热度替代真实爬虫数据

## 7. 算法建议

先用简单、可解释、便于答辩的规则模型：

- 技能匹配权重：40%
- 兴趣匹配权重：25%
- 性格匹配权重：20%
- 专业背景权重：10%
- 市场热度权重：5%

输出要求：

- 每个职业返回 0-100 分
- 返回影响最大的 3 个原因
- 返回推荐的主推/副推方向

## 8. 演示数据要求

为了答辩稳定，建议内置 3 组演示用户：

1. 偏技术型用户
2. 偏产品/沟通型用户
3. 偏数据分析型用户

这样你可以快速展示不同画像下的推荐差异。

## 9. 验收标准

第一阶段完成的最低验收标准：

1. 本地一键启动成功
2. 首页、测评、报告、选择流程可完整走通
3. 匹配结果不是写死文案，而是根据输入变化
4. 线上链接可访问
5. GitHub 仓库代码与线上版本一致
6. 截图覆盖关键页面

## 10. GitHub 与部署策略

建议把 GitHub 作为唯一事实源。

工作流建议：

1. 本地开发并提交到 Git
2. 推送到 GitHub
3. 部署平台连接 GitHub 主分支自动部署
4. 每次更新直接推送，线上自动同步

注意：

- 不要把“永久分享链接”理解成 GitHub 代码链接
- 分享链接应该是正式部署地址，不是 localhost，也不是临时预览地址

## 11. 推荐工具组合

### 必备

- Git
- GitHub
- 一个稳定部署平台：Vercel / Netlify / Render
- 浏览器自动化截图工具：Playwright

### 按场景选择

- 纯前端项目：Vercel 或 Netlify
- 前后端项目：Render
- ASP.NET Core：优先补齐 .NET SDK、EF Core 工具链

## 12. 给 Codex 的总控提示词

下面这段可以直接作为长期主提示词使用。

```text
你现在是这个课程项目的主开发代理。请基于仓库中的 PRD 文档
`人才技能量化系统·人才素描模块PRD.md`
持续迭代实现“人才素描模块”。

项目目标：
1. 本地可运行
2. GitHub 可持续同步
3. 具备长期有效的线上分享链接
4. 能输出截图、代码、分享链接三类提交材料

开发原则：
1. 先完成可演示 MVP，再逐步扩展
2. 优先实现真实可运行代码，不要只写伪代码
3. 如果后端或外部数据暂不具备，先用 mock 数据和可替换结构
4. 每次改动后同步更新 README、提交说明和必要截图说明
5. 所有实现都要尽量与 PRD 对齐，但允许分阶段交付

你需要优先完成：
1. 设计清晰的目录结构
2. 搭建本地开发环境
3. 实现首次测评 -> 匹配计算 -> 报告展示 -> 职业方向选择 的完整闭环
4. 预留后续趋势分析、提醒、月报、职业标签库的扩展点
5. 给出部署方案，并保证后续 GitHub 推送后能自动更新线上版本

输出要求：
1. 直接修改仓库代码，而不是只给建议
2. 每次说明改动内容、运行方式、验证方式
3. 如果发现 PRD 存在模糊点，做合理默认并写入文档
4. 尽量补充测试数据、空状态、错误处理、演示用种子数据
```

## 13. 给 Codex 的单轮迭代提示词

```text
请基于当前仓库继续迭代“人才素描模块”，这一次只处理以下目标：

[在这里填本轮目标，例如：完成首次测评问卷页面和匹配计算逻辑]

要求：
1. 先检查现有代码和文档，不要重复造轮子
2. 保持可以本地运行
3. 本轮改动完成后，补充 README 或 docs 中必要说明
4. 如果涉及 UI，请保证桌面端展示清晰，适合截图提交
5. 如果缺少真实后端或数据源，可以先用 mock，但结构必须可替换
6. 完成后告诉我：
   - 改了哪些文件
   - 如何运行
   - 如何验证
   - 下一轮最合理的开发项是什么
```

## 14. 给 Codex 的发布提示词

```text
请把当前项目整理到“可提交/可演示”状态，并完成以下事情：

1. 检查本地是否能正常启动
2. 检查 README 是否包含运行方式、项目简介、部署地址占位
3. 检查是否具备适合截图的关键页面
4. 输出一份提交材料清单
5. 如仓库已连接 GitHub，请给出最合理的部署方式
6. 如项目已部署，请说明后续如何保持 GitHub 与线上同步更新

注意：
1. 不要破坏现有代码
2. 优先补齐提交所需材料
3. 发现缺口时直接修补
```

## 15. 开发节奏建议

推荐每轮只做一个明确目标：

1. 初始化工程
2. 首次测评页面
3. 匹配算法与报告页
4. 职业方向选择与首页卡片
5. 趋势图和提醒
6. 部署与截图
7. README 和答辩材料整理
